Change ID: 2
Title: Clear upload file buffers after pipeline completes (memory leak fix)
Problem: uploadedFiles Map in routes/upload.ts was never pruned. Every file uploaded
         stayed in memory until server restart. Two 20 MB PDFs per session = 40 MB
         accumulated indefinitely. Under any sustained use this would OOM the server.
Fix: Added finally block after try/catch in the /analyze handler. Iterates fileIds
     and calls uploadedFiles.delete(id) for each. finally guarantees cleanup on all
     three exit paths: success, pipeline error (422), and thrown exception (500).
Files Affected:
  - server/src/routes/upload.ts
Code Changes:
  Added after catch block:
    } finally {
      // Release file buffers regardless of outcome — prevents memory accumulation
      for (const id of fileIds) uploadedFiles.delete(id)
    }
Impact: Memory is released immediately after each analysis session regardless of
        outcome. Map size stays at zero between requests.
Risk: Low — finally runs after response is sent. No change to response shape,
      pipeline logic, or upstream/downstream contracts.

Change ID: 4
Title: Unify CaseData schema — replace server-schema with StoredCase matching pipeline output
Problem: Two incompatible CaseData types existed in the same codebase:
         - server/src/data/mockData.ts used a server-schema (patientName, summary.totalBilled,
           BillIssue with amount/documentId/actionIds, etc.)
         - src/data/mockCase.ts used a frontend-schema (flat totalBilled, Issue with
           relatedDocIds/explanation, Action with whyItMatters/draft/script)
         outputMapper.ts (Person C boundary) produced frontend-schema. The cases CRUD
         stored server-schema. These were structurally incompatible — any attempt by
         the frontend to use /api/cases data would silently receive the wrong shape.
         stats.ts accessed c.summary.totalBilled which TypeScript now correctly rejects.
Fix: Replaced mockData.ts entirely with StoredCase interface that exactly matches
     what outputMapper.ts produces, plus status and createdAt server-only fields.
     Updated store.ts to use StoredCase throughout.
     Updated cases.ts route to use StoredCase — POST handler now uses flat fields.
     Updated stats.ts to use c.totalBilled (flat) instead of c.summary.totalBilled.
     Updated upload.ts to import StoredCase and removed the as-any cast from FIX-3.
     MOCK_STATS dead export removed as part of the mockData.ts rewrite.
     store.update() now strips id and createdAt from patches (immutable fields).
     Demo seed data (DEMO_CASES) rewritten to match the new StoredCase shape.
Files Affected:
  - server/src/data/mockData.ts   (full rewrite — new types + new demo data)
  - server/src/data/store.ts      (CaseData → StoredCase, immutable field guard)
  - server/src/routes/cases.ts    (CaseData → StoredCase, POST handler updated)
  - server/src/routes/stats.ts    (c.summary.* → c.* flat fields, totalSaved removed)
  - server/src/routes/upload.ts   (import StoredCase, as-any cast → as StoredCase)
Impact: Single source of truth for case shape across pipeline output, store, and
        CRUD routes. /api/cases now returns data the frontend can consume directly.
        /api/stats correctly aggregates real cases from the pipeline.
        TypeScript will catch any future shape divergence at compile time.
Risk: Medium — touches 5 files. Demo seed data shape changed (non-breaking for
